# AI Daily Digest Agent ŌĆö Implementation Plan

## Context

Build a Managed Agent that runs daily at 9am ET, searches for the latest AI news (Anthropic, OpenAI, Google AI, developer tools/vibe coding), pushes a detailed daily note to a GitHub-backed Obsidian vault, and sends a bullet-point summary to Slack via webhook.

**Repos:**
- `ai-daily-digest` ŌĆö Obsidian vault (GitHub resource, agent pushes notes)
- `ai-digest-orchestrator` ŌĆö Orchestrator code + GitHub Actions workflow (currently empty except LICENSE)

**Secrets already configured in `ai-digest-orchestrator`:**
- `ANTHROPIC_API_KEY`
- `GIT_PAT`
- `SLACK_WEBHOOK_URL`

---

## Architecture

```
GitHub Actions cron (9am ET daily)
  ŌåÆ run.py (orchestrator)
      ŌåÆ sessions.create(agent=AGENT_ID, env=ENV_ID, resources=[obsidian repo])
      ŌåÆ events.send(user.message: "fetch last 24hr AI news")
      ŌåÆ stream events:
          agent searches web (web_search + web_fetch)
          agent writes daily note markdown
          agent commits + pushes to ai-daily-digest via the GitHub MCP toolset
          agent emits custom_tool_use: send_slack_message
      ŌåÆ orchestrator POSTs to Slack webhook
      ŌåÆ session archives
```

## Files to Create (in `ai-digest-orchestrator` repo)

### 1. `setup.py` ŌĆö One-time setup (idempotent ŌĆö safe to re-run)

Creates the environment and agent, prints IDs to save as GitHub secrets.

- **Idempotency with drift detection**: Before creating, **paginate** `environments.list()` / `agents.list()` until exhausted ŌĆö do not trust a single page. Filter to non-archived entries with matching names (`ai-digest-env` / `AI Daily Digest`). If multiple matches exist, sort deterministically by `created_at` descending; pick the newest as the candidate. **Do not archive the extras by default** (destructive) ŌĆö log a WARN listing their IDs and require `--prune-duplicates` to actually archive them.
- **Canonical config diff ŌĆö applied to BOTH agent and environment**, to avoid spurious recreate churn from API-normalized orderings / server-added defaults, and to catch env drift like networking mode flipping under our feet.

All canonicalization uses a single shared helper for shape tolerance ŌĆö the SDK returns objects in some paths and dicts in others, so avoid `.get()`/`.attr` directly:

```python
def _get(obj, key, default=None):
    if obj is None: return default
    if isinstance(obj, dict): return obj.get(key, default)
    return getattr(obj, key, default)

def _model_id(m):
    if isinstance(m, str): return m
    v = _get(m, "id")
    if v is not None: return v
    raise RuntimeError(f"cannot extract model id from shape {type(m).__name__!r}: {m!r}")

def _sorted_if_list(v):
    return sorted(v) if isinstance(v, list) else v
```

(No `str(m)` fallback for model ŌĆö default `__repr__` may include memory addresses, which would change between process runs and force spurious recreates.)

```python
def canonical_agent(a):
    servers = [{"name": _get(s,"name"), "url": _get(s,"url"), "type": _get(s,"type")}
               for s in (_get(a, "mcp_servers") or [])]
    tools = [canonical_tool(t) for t in (_get(a, "tools") or [])]
    return {
        "model": _model_id(_get(a, "model")),
        "system": (_get(a, "system") or "").strip(),
        "mcp_servers": sorted(servers, key=lambda s: s["name"] or ""),
        "tools": sorted(tools, key=lambda t: (t.get("type",""), t.get("name",""))),
    }

def canonical_env(e):
    cfg = _get(e, "config") or {}
    net = _get(cfg, "networking") or {}
    return {
        "config.type": _get(cfg, "type"),
        "networking.type": _get(net, "type"),
        "allowed_hosts": _sorted_if_list(_get(net, "allowed_hosts", []) or []),
        "packages": {k: _sorted_if_list(v) for k, v in (_get(cfg, "packages", {}) or {}).items()},
    }
```

`canonical_tool(t)` normalizes `enabled` flags and `default_config.permission_policy.type` using the same `_get` helper. Drop any field the API adds that we never set (e.g. `id`, `created_at`, `object`, `archived_at`).

Compute `sha256(json.dumps(canonical, sort_keys=True, default=str))` for both sides and compare. Hashes match ŌåÆ reuse. Hashes differ ŌåÆ log which top-level field's sub-hash differs, then **recreate** (archive the old, create a new one). Reuse silently baking in stale config is the failure mode we're defending against. `--force` always creates fresh without comparing.
- **Environment**: `ai-digest-env`, `config: { type: "cloud", networking: { type: "unrestricted" } }`. Environments do **not** accept user-defined env vars or secrets ŌĆö GIT_PAT is supplied per-session via the `github_repository` resource's `authorization_token`, and gets rotated on each run.
- **Agent config**:
  - Name: `AI Daily Digest`
  - Model: `claude-sonnet-4-6`
  - System prompt: see below
  - **No `mcp_servers`** ŌĆö the original design used the GitHub Copilot MCP (`api.githubcopilot.com/mcp/`) for writes, but that endpoint requires a GitHub Copilot subscription, which is not available on this account. All writes move to the orchestrator side instead.
  - Tools:
    - Agent toolset ŌĆö exact shape:
      ```python
      {
          "type": "agent_toolset_20260401",
          "configs": [
              {"name": "write", "enabled": False},
              {"name": "edit", "enabled": False},
          ],
      }
      ```
      This leaves `bash`, `read`, `grep`, `glob`, `web_search`, `web_fetch` enabled. The agent reads the mounted repo to check prior notes; all writes go through the `write_daily_note` custom tool handled by the orchestrator.
    - Custom tool `write_daily_note` with input schema `{content: string, content_sha256: string}` ŌĆö the agent calls this with the full markdown body and its SHA-256. The orchestrator commits the file via the GitHub REST API and returns the commit SHA (or a no-op result if the content is byte-identical to what's already on main).
    - Custom tool `send_slack_message` with input schema `{summary: string}` ŌĆö unchanged.
    - (No `mcp_toolset` entry ŌĆö nothing to configure permissions for.)
- Outputs `AGENT_ID` and `ENVIRONMENT_ID` ŌåÆ user adds these as GitHub Actions secrets

```

2. A short bullet-point summary for Slack

## Sources to check

- OpenAI blog and announcements
- Google AI / DeepMind / Gemini announcements

## Workflow
1. Use web_search to find recent AI news from each source category
2. Use web_fetch to get details from relevant articles
3. Optionally read the mounted repo at `/workspace/ai-daily-digest` (via `read`/`bash`) to see yesterday's note or check whether today's file already exists. Read-only ŌĆö do NOT attempt to write or push from bash.
4. Compose the full markdown daily note content in memory.
5. Compute the SHA-256 of the UTF-8-encoded content (use a small bash one-liner like `sha256sum` or the `hashlib` equivalent via the agent's Python tool if available).
6. Call **`write_daily_note`** with `{"content": "<full markdown>", "content_sha256": "<64-hex>"}`. The orchestrator will:
   - Verify the claimed hash matches the content it received.
   - Commit/create the file at `YYYY/MM/YYYY-MM-DD.md` on `main` via the GitHub REST API.
   - Return `{"committed": true, "commit_sha": "...", "no_op": false}` on a real write, or `{"committed": false, "no_op": true}` if today's file was already byte-identical.
   - Return `{"committed": false, "error": "..."}` on failure.
7. Call `send_slack_message` with a short bullet-point summary. **Always** fire ŌĆö even on no-op reruns. Duplicate Slack messages on reruns are allowed by design.
```

### 2. `run.py` ŌĆö Runtime script (runs every invocation)

Load AGENT_ID, ENVIRONMENT_ID, GIT_PAT, SLACK_WEBHOOK_URL from env vars
ŌåÆ client = anthropic.Anthropic()
ŌåÆ session = client.beta.sessions.create(
      agent=AGENT_ID,
      environment_id=ENVIRONMENT_ID,
      resources=[{
          type: "github_repository",
          url: "https://github.com/ZH-L1N/ai-daily-digest",
          authorization_token: GIT_PAT,
          mount_path: "/workspace/ai-daily-digest",
          checkout: {type: "branch", name: "main"}
      }]
  )
ŌåÆ Open stream (stream-first pattern)
ŌåÆ Compute `path` **once, before sending the kickoff message** (not recomputed later), using `America/New_York`:
  ```python
  now_et = datetime.now(ZoneInfo("America/New_York"))
  date_str = now_et.strftime("%Y-%m-%d")
  path = f"{now_et.strftime('%Y')}/{now_et.strftime('%m')}/{date_str}.md"
  run_start_utc = datetime.now(timezone.utc)
  before_sha = ...  # GET contents at {path}?ref=main, None if 404
  ```
  This single `path` is reused for the kickoff message AND the post-loop verification ŌĆö so even if the run spans midnight ET, both sides agree on the same "today." (Placement-wise, these lines can live anywhere before `events.send(user.message=...)`; what matters is the relative ordering, not that they sit at module top.)
ŌåÆ Send kickoff message using f-string interpolation of those runtime variables (the literal values below are illustrative ŌĆö in code they are dynamically formatted from `date_str` and `path`):
  ```python
  kickoff = f"""\
  Today's date (America/New_York): {date_str}
  Target file path (must use exactly this path): {path}
  Branch: main
  Fetch the last 24 hours of AI news, write today's digest to {path} in the Obsidian vault via the GitHub MCP tools, and send the Slack summary. End your final message with CONTENT_SHA256: <sha256 hex of the file's UTF-8 bytes>.
  """
  ```
  (NO secrets in the prompt ŌĆö GIT_PAT is passed per-session via the `github_repository` resource's `authorization_token` and powers the GitHub MCP writes; the Slack webhook is held orchestrator-side and never shown to the agent.)
ŌåÆ Track `slack_sent = False`, `slack_error = None`, `note_committed = False`, `last_claimed_sha = None`, `pending_custom_tool_uses = set()` (event IDs seen but not yet answered), `answered_custom_tool_uses = set()` (event IDs we've already responded to ŌĆö needed because a queued `session.status_idle` may still reference an event we already answered)
ŌåÆ Event loop:
    - agent.message ŌåÆ log text
    - session.error (session-level runtime/auth error event) ŌåÆ log the full error payload and raise SystemExit(1).
    - agent.custom_tool_use ŌåÆ dispatch on `name`:
      - `write_daily_note` ŌåÆ handle per "Custom tool handling (write_daily_note)" below.
      - `send_slack_message` ŌåÆ handle per "Custom tool handling (send_slack_message)" below.
      - anything else ŌåÆ return `is_error=true` with message `"unknown tool"`.
    - session.status_idle (end of a turn) ŌåÆ handle per "Idle turn handling" detail below.
    - session.status_terminated ŌåÆ log and raise SystemExit(1)
ŌåÆ After loop:
    - If not `note_committed` ŌåÆ log and raise SystemExit(1) (agent never called `write_daily_note`, or every call failed).
    - If not `slack_sent` ŌåÆ log `slack_error` and raise SystemExit(1).
    - **Deterministic success check** ŌĆö since the orchestrator performed the write itself, success is mostly proven by the tool-result bookkeeping. One final belt-and-suspenders check: GET the file from `main` and confirm its SHA-256 matches the last `content_sha256` the agent supplied. This catches the rare case where a concurrent force-push rewrote main between our commit and this check.
      ```python
      after_sha, observed_bytes = github_get_contents(git_pat, path)
      if after_sha is None or hashlib.sha256(observed_bytes).hexdigest() != last_claimed_sha:
          raise SystemExit(1)
      ```
    - All GitHub requests use `Authorization: Bearer {GIT_PAT}`, `Accept: application/vnd.github+json`, `User-Agent: ai-digest-orchestrator`, via `urllib.request` (stdlib) with an explicit `timeout=10` per read, `timeout=30` per write.
ŌåÆ Always archive the session in a `finally` block, even on SystemExit or exception ŌĆö best-effort, swallow errors from archive itself so a failed archive doesn't mask the original failure reason

**Slack webhook payload:**
```json
{ "text": "<summary from agent>" }
```

**Custom tool handling (write_daily_note).** On an `agent.custom_tool_use` for `write_daily_note`:

1. Parse `{"content": str, "content_sha256": str}` from the tool input.
2. Verify `hashlib.sha256(content.encode("utf-8")).hexdigest() == content_sha256`. If not, return a `is_error=true` custom_tool_result with `{"committed": false, "error": "hash mismatch"}` ŌĆö do NOT commit. Do not raise ŌĆö let the agent retry.
3. GET the current file at `path` on `main`. If it exists and its decoded bytes already equal `content.encode("utf-8")`, skip the commit and return `{"committed": false, "no_op": true}` ŌĆö this is the byte-identical no-op path.
4. Otherwise PUT to `/repos/ZH-L1N/ai-daily-digest/contents/{path}` with body:
   ```json
   {
     "message": "Daily digest YYYY-MM-DD",
     "content": "<base64-encoded UTF-8 bytes>",
     "branch": "main",
     "sha": "<before_sha if updating, omit if creating>"
   }
   ```
   Use `Authorization: Bearer {GIT_PAT}`, explicit `timeout=30` (file writes can be slower than reads). Return `{"committed": true, "commit_sha": <response.commit.sha>, "no_op": false}` to the agent as a non-error custom_tool_result.
5. On network/HTTP error: return `is_error=true` with `{"committed": false, "error": "<detail>"}`. Do not raise ŌĆö the agent may retry.
6. Track orchestrator-side: `note_committed = (result.committed or result.no_op)` and stash the `content_sha256` for the final verification.

**Custom tool handling (send_slack_message).** On an `agent.custom_tool_use` for `send_slack_message`:

1. Add `event.id` to `pending_custom_tool_uses`.
2. Try POST to `SLACK_WEBHOOK_URL` with `{"text": <summary>}` using `urllib.request` (stdlib), `timeout=10`, and two retries on non-2xx or network errors with 2s/4s backoff before declaring failure.
3. On success: set `slack_sent = True`, then send a custom_tool_result with `is_error=False`, content text `"sent"`.
4. On final failure (after retries): set `slack_error = <detail>`, send a custom_tool_result with `is_error=True` and the error message. Do NOT raise ŌĆö the agent may retry.
5. After sending the result, move the event id from `pending_custom_tool_uses` to `answered_custom_tool_uses`.

Send the tool result exactly like this ŌĆö and if the POST itself fails (network flap, Anthropic 5xx), raise SystemExit(1) immediately rather than swallow (a silent tool-result drop hangs the agent at `requires_action` until workflow timeout):

```python
try:
    client.beta.sessions.events.send(
        session.id,
        events=[{
            "type": "user.custom_tool_result",
            "custom_tool_use_id": custom_tool_use_event.id,
            "content": [{"type": "text", "text": result_text}],
            "is_error": is_error,
        }],
    )
except Exception as e:
    log.error("failed to POST custom_tool_result id=%s: %s", custom_tool_use_event.id, e)
    raise SystemExit(1)
```

**Idle turn handling.** On `session.status_idle`, inspect `stop_reason`. The SDK may model it as a string or an object with a `type` field ŌĆö normalize first:

```python
reason_type = stop_reason if isinstance(stop_reason, str) else _get(stop_reason, "type")
```

(`_get` is the same shape-tolerant helper defined in the setup.py canonicalization section above ŌĆö reuse it here.)

- `reason_type == "end_turn"` ŌåÆ success, break out of the event loop.
- `reason_type == "requires_action"` ŌåÆ extract blocking event IDs, tolerating both nested (`.requires_action.event_ids`) and flat (`.event_ids`) paths:

```python
blocking = (
    _get(_get(stop_reason, "requires_action"), "event_ids")
    or _get(stop_reason, "event_ids")
    or []
)
```

  - If `blocking` is non-empty and is a subset of `pending_custom_tool_uses Ōł¬ answered_custom_tool_uses`, continue the loop (we already responded or are about to ŌĆö the union guards the race where a queued `status_idle` references an id we've answered).
  - If `blocking` is empty under `requires_action`, that's a schema-path miss ŌĆö log the full raw `stop_reason` and raise SystemExit(1). Do NOT loop on empty blocking, or we'll hang until workflow timeout.
  - Otherwise, `blocking` has IDs we don't recognize (can happen when `status_idle` is delivered before its corresponding `agent.custom_tool_use` arrived on the stream). Before failing, **paginate** `client.beta.sessions.events.list(session.id)` until either every unknown id appears and is a custom_tool_use we can respond to (loop back and handle it) or the pagination is exhausted. Only raise SystemExit(1) after full pagination still leaves unknown blocking ids.
- any other (`retries_exhausted`, `max_turns`, `refusal`, `error`, ŌĆ”) ŌåÆ log full session + last events and raise SystemExit(1).

## Resolved Questions

- **Timezone**: Eastern Time, year-round ŌĆö workflow uses `cron: '13 9 * * *'` with `timezone: 'America/New_York'` (GitHub Actions supports this; DST is handled automatically; `:13` dodges top-of-hour scheduler congestion)
- **GitHub username**: `ZH-L1N`
- **Slack duplicates on reruns**: **Allowed**. Every run (scheduled or `workflow_dispatch`) posts a fresh Slack message. No de-dup state tracking.
- **No-op reruns (byte-identical content)**: **Skip the commit, but still send Slack.** Verification is machine-checkable: the agent emits `CONTENT_SHA256: <hex>` in its final message (the single contract ŌĆö no free-form sentinels). Orchestrator fetches today's file from GitHub, computes its SHA-256, and accepts the no-op only if the observed hash equals the agent-claimed hash. Slack always fires so the user sees the pipeline ran.

## Failure modes & diagnosis

- **Agent never calls `write_daily_note`**: `note_committed` stays False and the post-loop check fails. Inspect the stream for `agent.message` entries to see what the agent did instead; usually indicates a system-prompt regression.
- **Agent supplies wrong SHA-256 in `write_daily_note`**: the tool handler rejects with `is_error=true, "hash mismatch"`. Agent normally retries once. Repeated failures suggest the agent is miscomputing ŌĆö add explicit hashing instructions to the system prompt.
- **Session hangs at `requires_action` with no resolvable blocking IDs**: workflow `timeout-minutes: 15` kills the run; `run.py`'s robust extractor also raises SystemExit(1) proactively.
- **Slack webhook down**: agent logs the error via `is_error=true` tool result; `run.py` retries twice with 2s/4s backoff, then raises SystemExit(1) at end of loop so the workflow turns red.
- **Scheduled workflow auto-disabled**: GitHub automatically disables scheduled workflows on repos with no commit activity for 60 days. If the digest stops firing on schedule but `workflow_dispatch` still works, check **Actions ŌåÆ AI Daily Digest ŌåÆ Enable workflow**. Prevent by ensuring the orchestrator repo sees at least one commit every ~50 days (or wire a tiny scheduled no-op commit).
