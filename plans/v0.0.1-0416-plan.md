# AI Daily Digest Agent — Implementation Plan

> **Snapshot**: v0.0.1 — 2026-04-16. Frozen copy of the approved plan used to generate `setup.py`, `run.py`, `requirements.txt`, and `.github/workflows/daily-digest.yml`.

## Context

Build a Managed Agent that runs daily at 9am ET, searches for the latest AI news (Anthropic, OpenAI, Google AI, developer tools/vibe coding), pushes a detailed daily note to a GitHub-backed Obsidian vault, and sends a bullet-point summary to Slack via webhook.

**Repos:**
- `ai-daily-digest` — Obsidian vault (GitHub resource, agent pushes notes)
- `ai-digest-orchestrator` — Orchestrator code + GitHub Actions workflow (currently empty except LICENSE)

**Secrets already configured in `ai-digest-orchestrator`:**
- `ANTHROPIC_API_KEY`
- `GIT_PAT`
- `SLACK_WEBHOOK_URL`

---

## Architecture

```
GitHub Actions cron (9am ET daily)
  → run.py (orchestrator)
      → sessions.create(agent=AGENT_ID, env=ENV_ID, resources=[obsidian repo])
      → events.send(user.message: "fetch last 24hr AI news")
      → stream events:
          agent searches web (web_search + web_fetch)
          agent writes daily note markdown
          agent commits + pushes to ai-daily-digest via the GitHub MCP toolset
          agent emits custom_tool_use: send_slack_message
      → orchestrator POSTs to Slack webhook
      → session archives
```

## Files to Create (in `ai-digest-orchestrator` repo)

### 1. `setup.py` — One-time setup (idempotent — safe to re-run)

Creates the environment and agent, prints IDs to save as GitHub secrets.

- **Idempotency with drift detection**: Before creating, **paginate** `environments.list()` / `agents.list()` until exhausted — do not trust a single page. Filter to non-archived entries with matching names (`ai-digest-env` / `AI Daily Digest`). If multiple matches exist, sort deterministically by `created_at` descending; pick the newest as the candidate. **Do not archive the extras by default** (destructive) — log a WARN listing their IDs and require `--prune-duplicates` to actually archive them.
- **Canonical config diff — applied to BOTH agent and environment**, to avoid spurious recreate churn from API-normalized orderings / server-added defaults, and to catch env drift like networking mode flipping under our feet.

All canonicalization uses a single shared helper for shape tolerance — the SDK returns objects in some paths and dicts in others, so avoid `.get()`/`.attr` directly:

```python
def _get(obj, key, default=None):
    if obj is None: return default
    if isinstance(obj, dict): return obj.get(key, default)
    return getattr(obj, key, default)

def _model_id(m):
    if isinstance(m, str): return m
    v = _get(m, "id")
    if v is not None: return v
    raise RuntimeError(f"cannot extract model id from shape {type(m).__name__!r}: {m!r}")

def _sorted_if_list(v):
    return sorted(v) if isinstance(v, list) else v
```

(No `str(m)` fallback for model — default `__repr__` may include memory addresses, which would change between process runs and force spurious recreates.)

```python
def canonical_agent(a):
    servers = [{"name": _get(s,"name"), "url": _get(s,"url"), "type": _get(s,"type")}
               for s in (_get(a, "mcp_servers") or [])]
    tools = [canonical_tool(t) for t in (_get(a, "tools") or [])]
    return {
        "model": _model_id(_get(a, "model")),
        "system": (_get(a, "system") or "").strip(),
        "mcp_servers": sorted(servers, key=lambda s: s["name"] or ""),
        "tools": sorted(tools, key=lambda t: (t.get("type",""), t.get("name",""))),
    }

def canonical_env(e):
    cfg = _get(e, "config") or {}
    net = _get(cfg, "networking") or {}
    return {
        "config.type": _get(cfg, "type"),
        "networking.type": _get(net, "type"),
        "allowed_hosts": _sorted_if_list(_get(net, "allowed_hosts", []) or []),
        "packages": {k: _sorted_if_list(v) for k, v in (_get(cfg, "packages", {}) or {}).items()},
    }
```

`canonical_tool(t)` normalizes `enabled` flags and `default_config.permission_policy.type` using the same `_get` helper. Drop any field the API adds that we never set (e.g. `id`, `created_at`, `object`, `archived_at`).

Compute `sha256(json.dumps(canonical, sort_keys=True, default=str))` for both sides and compare. Hashes match → reuse. Hashes differ → log which top-level field's sub-hash differs, then **recreate** (archive the old, create a new one). Reuse silently baking in stale config is the failure mode we're defending against. `--force` always creates fresh without comparing.
- **Environment**: `ai-digest-env`, `config: { type: "cloud", networking: { type: "unrestricted" } }`. Environments do **not** accept user-defined env vars or secrets — GIT_PAT is supplied per-session via the `github_repository` resource's `authorization_token`, and gets rotated on each run.
- **Agent config**:
  - Name: `AI Daily Digest`
  - Model: `claude-sonnet-4-6`
  - System prompt: see below
  - `mcp_servers`: `[{type: "url", name: "github", url: "https://api.githubcopilot.com/mcp/"}]` — the GitHub MCP server handles all repo writes (commit/push). Auth is taken from the session's `github_repository.authorization_token`, not declared here.
  - Tools:
    - Agent toolset — exact shape (per the Python SDK examples I fetched):
      ```python
      {
          "type": "agent_toolset_20260401",
          "configs": [
              {"name": "write", "enabled": False},
              {"name": "edit", "enabled": False},
          ],
      }
      ```
      This leaves `bash`, `read`, `grep`, `glob`, `web_search`, `web_fetch` enabled. Repo mutations go through the GitHub MCP toolset; disabling local-write tools is defense-in-depth against the agent choosing a non-MCP write path. Default `permission_policy` is `always_allow` per the docs — no override needed.
    - `{type: "mcp_toolset", mcp_server_name: "github", default_config: {permission_policy: {type: "always_allow"}}}` — **critical**: MCP toolsets default to `always_ask`, which would emit `requires_action` waiting for `user.tool_confirmation` on every GitHub MCP call. This override runs GitHub MCP tools automatically. We trust this MCP server because we control its auth token.
    - Custom tool: `send_slack_message` with input schema `{summary: string}` (custom tools are always client-executed — permission policies don't apply to them)
- Outputs `AGENT_ID` and `ENVIRONMENT_ID` → user adds these as GitHub Actions secrets

**System prompt (draft):**

```
You are an AI news curator. Your job is to find the most important AI news from the last 24 hours and produce two outputs:

1. A detailed daily note (Markdown) saved to the Obsidian vault
2. A short bullet-point summary for Slack

## Sources to check
- Anthropic blog and announcements
- OpenAI blog and announcements
- Google AI / DeepMind / Gemini announcements
- AI developer tools news (Claude Code, Cursor, Windsurf, Copilot, Replit, v0, Bolt, etc.)

## What to include
- New product launches and features
- Major updates to existing tools
- New developer-facing capabilities
- Anything relevant to "vibe coding" and AI-assisted development

## What to exclude
- Academic papers and research unless they ship a product
- Rumors or speculation
- Funding/hiring news unless it directly affects a product

## Workflow
1. Use web_search to find recent AI news from each source category
2. Use web_fetch to get details from relevant articles
3. Compose the markdown daily note content in memory (target path: `YYYY/MM/YYYY-MM-DD.md` in the `ai-daily-digest` repo on `main`)
4. Use the **GitHub MCP tools** (exposed via `mcp_toolset`, server name `github`) to commit and push the file — do NOT use bash+git for writes. You may read the mounted repo at `/workspace/ai-daily-digest` with `read`/`bash` to check whether today's file already exists. (Structural enforcement: GIT_PAT is only attached to the session's `github_repository` resource and the GitHub MCP server; it is not exposed as a sandbox env var, so plain `git push` from bash will fail on auth even if attempted.)
   - If file doesn't exist: create it via the MCP `create_or_update_file`-style tool with commit message `Daily digest YYYY-MM-DD`
   - If file exists (same-day rerun): update it via the same tool with the fresh content; skip the write entirely if the new content is byte-identical to the existing content (avoid empty commits)
5. Call send_slack_message with a short bullet-point summary. **Always** fire — even on same-day reruns and even when no commit was produced (byte-identical no-op). Duplicate Slack messages on reruns are allowed by design.
6. In your final message, **always** include a single line of the form `CONTENT_SHA256: <64-hex-chars>` computed over the UTF-8 bytes of today's intended note content (exactly what you wrote, or would have written for a no-op). The orchestrator independently fetches today's file from GitHub, computes its SHA-256, and compares — this is machine-verifiable proof of what's on main vs. what you intended. No free-form sentinels; the hash is the contract.

```

### 2. `run.py` — Runtime script (runs every invocation)

Load AGENT_ID, ENVIRONMENT_ID, GIT_PAT, SLACK_WEBHOOK_URL from env vars
→ client = anthropic.Anthropic()
→ session = client.beta.sessions.create(
      agent=AGENT_ID,
      environment_id=ENVIRONMENT_ID,
      resources=[{
          type: "github_repository",
          url: "https://github.com/ZH-L1N/ai-daily-digest",
          authorization_token: GIT_PAT,
          mount_path: "/workspace/ai-daily-digest",
          checkout: {type: "branch", name: "main"}
      }]
  )
→ Open stream (stream-first pattern)
→ Compute `path` **once, before sending the kickoff message** (not recomputed later), using `America/New_York`:
  ```python
  now_et = datetime.now(ZoneInfo("America/New_York"))
  date_str = now_et.strftime("%Y-%m-%d")
  path = f"{now_et.strftime('%Y')}/{now_et.strftime('%m')}/{date_str}.md"
  run_start_utc = datetime.now(timezone.utc)
  before_sha = ...  # GET contents at {path}?ref=main, None if 404
  ```
  This single `path` is reused for the kickoff message AND the post-loop verification — so even if the run spans midnight ET, both sides agree on the same "today." (Placement-wise, these lines can live anywhere before `events.send(user.message=...)`; what matters is the relative ordering, not that they sit at module top.)
→ Send kickoff message using f-string interpolation of those runtime variables (the literal values below are illustrative — in code they are dynamically formatted from `date_str` and `path`):
  ```python
  kickoff = f"""\
  Today's date (America/New_York): {date_str}
  Target file path (must use exactly this path): {path}
  Branch: main
  Fetch the last 24 hours of AI news, write today's digest to {path} in the Obsidian vault via the GitHub MCP tools, and send the Slack summary. End your final message with CONTENT_SHA256: <sha256 hex of the file's UTF-8 bytes>.
  """
  ```
  (NO secrets in the prompt — GIT_PAT is passed per-session via the `github_repository` resource's `authorization_token` and powers the GitHub MCP writes; the Slack webhook is held orchestrator-side and never shown to the agent.)
→ Track `slack_sent = False`, `slack_error = None`, `pending_custom_tool_uses = set()` (event IDs seen but not yet answered), `answered_custom_tool_uses = set()` (event IDs we've already responded to — needed because a queued `session.status_idle` may still reference an event we already answered)
→ Track `mcp_errors_seen = []` (accumulates `agent.mcp_tool_result` entries where `is_error=true`)
→ Event loop:
    - agent.message → log text
    - agent.mcp_tool_result → if `is_error=true`, append `{tool_use_id, content}` to `mcp_errors_seen` and log a WARN. Don't fail immediately — the agent may recover via retry.
    - session.error (session-level runtime/auth error event) → log the full error payload and raise SystemExit(1). This catches MCP auth failures and other fatal server-side problems that could otherwise let the run drift to a false-green.
    - agent.custom_tool_use (send_slack_message) → handle per the "Custom tool handling" detail below.
    - session.status_idle (end of a turn) → handle per "Idle turn handling" detail below.
    - session.status_terminated → log and raise SystemExit(1)
→ After loop:
    - If not `slack_sent` → log `slack_error` and raise SystemExit(1)
    - **Deterministic success check** — machine-verifiable, no trust in agent self-reports.
      - **Reuse** the single `path` variable captured before kickoff (see earlier step — `path = "{yyyy}/{mm}/{yyyy}-{mm}-{dd}.md"` from ET-now). Do **not** recompute today's date here; if the run crossed midnight ET, recomputing would validate a different file than the one the agent was instructed to write against.
      - **Before kickoff**, capture `run_start_utc = datetime.now(timezone.utc)` (UTC for unambiguous comparison later) and `before_sha = GET /repos/ZH-L1N/ai-daily-digest/contents/{path}?ref=main`'s `sha` (None if 404).
      - **After the event loop**, extract `agent_claimed_hash`. The stream can emit many `agent.message` events; **accumulate every `agent.message`'s text in stream order into a single `all_text` string**, separated by newlines. Scan with `matches = re.findall(r"^CONTENT_SHA256:\s+([0-9a-fA-F]{64})\s*$", all_text, re.MULTILINE)` — `re.MULTILINE` so `^`/`$` match line boundaries (not whole-string), and the `[0-9a-fA-F]` character class accepts uppercase hex too (some tools emit uppercase). Take `agent_claimed_hash = matches[-1].lower()` (normalize to lowercase for comparison, since `hashlib.sha256().hexdigest()` always produces lowercase). Compare against `observed_hash` which is already lowercase by definition. If `matches` is empty, log and raise SystemExit(1) — the agent broke the contract.
      - Fetch the file now: `GET /repos/ZH-L1N/ai-daily-digest/contents/{path}?ref=main`. On 200, let `after_response` = parsed JSON, `after_sha = after_response["sha"]`, `observed_bytes = base64.b64decode(after_response["content"])`, `observed_hash = hashlib.sha256(observed_bytes).hexdigest()`. On 404: `after_sha = None`, `observed_hash = None` — **short-circuit to SystemExit(1) immediately** (rule 1 below will fail anyway, and skipping the commit lookup avoids a second 404 / empty-list parse).
      - Only if `after_sha is not None`: fetch the file's latest commit on main: `GET /repos/ZH-L1N/ai-daily-digest/commits?path={path}&per_page=1`. Guard with an explicit runtime check — **do not use `assert`** (it can be stripped under `python -O`): `commits = response_json; if not commits: log.error(...); raise SystemExit(1)`. Then `latest_commit_utc = datetime.fromisoformat(commits[0]["commit"]["committer"]["date"].replace("Z","+00:00"))` (GitHub returns RFC-3339 `Z` times; parse as tz-aware UTC).
      - **Accept success only if ALL of:**
        1. `after_sha is not None` (file exists on main).
        2. `observed_hash == agent_claimed_hash` (what's on main is byte-for-byte what the agent claims it intended).
        3. Either `latest_commit_utc >= run_start_utc` (this run actually wrote the commit), **or** `before_sha is not None and before_sha == after_sha` (legitimate no-op — file was already correct, independently verified by the hash match in rule 2).
      - Any failure → log `agent_claimed_hash`, `observed_hash`, `before_sha`, `after_sha`, `latest_commit_utc`, `run_start_utc`, and `mcp_errors_seen`, then raise SystemExit(1).
      - All GitHub requests use `Authorization: Bearer {GIT_PAT}`, `Accept: application/vnd.github+json`, `User-Agent: ai-digest-orchestrator`, via `urllib.request` (stdlib) with an explicit `timeout=10` per call.
→ Always archive the session in a `finally` block, even on SystemExit or exception — best-effort, swallow errors from archive itself so a failed archive doesn't mask the original failure reason

**Custom tool handling (send_slack_message).** On an `agent.custom_tool_use` for `send_slack_message`:

1. Add `event.id` to `pending_custom_tool_uses`.
2. Try POST to `SLACK_WEBHOOK_URL` with `{"text": <summary>}` using `urllib.request` (stdlib), `timeout=10`, and two retries on non-2xx or network errors with 2s/4s backoff before declaring failure.
3. On success: set `slack_sent = True`, then send a custom_tool_result with `is_error=False`, content text `"sent"`.
4. On final failure (after retries): set `slack_error = <detail>`, send a custom_tool_result with `is_error=True` and the error message. Do NOT raise — the agent may retry.
5. After sending the result, move the event id from `pending_custom_tool_uses` to `answered_custom_tool_uses`.

Send the tool result exactly like this — and if the POST itself fails (network flap, Anthropic 5xx), raise SystemExit(1) immediately rather than swallow (a silent tool-result drop hangs the agent at `requires_action` until workflow timeout):

```python
try:
    client.beta.sessions.events.send(
        session.id,
        events=[{
            "type": "user.custom_tool_result",
            "custom_tool_use_id": custom_tool_use_event.id,
            "content": [{"type": "text", "text": result_text}],
            "is_error": is_error,
        }],
    )
except Exception as e:
    log.error("failed to POST custom_tool_result id=%s: %s", custom_tool_use_event.id, e)
    raise SystemExit(1)
```

**Idle turn handling.** On `session.status_idle`, inspect `stop_reason`. The SDK may model it as a string or an object with a `type` field — normalize first:

```python
reason_type = stop_reason if isinstance(stop_reason, str) else _get(stop_reason, "type")
```

(`_get` is the same shape-tolerant helper defined in the setup.py canonicalization section above — reuse it here.)

- `reason_type == "end_turn"` → success, break out of the event loop.
- `reason_type == "requires_action"` → extract blocking event IDs, tolerating both nested (`.requires_action.event_ids`) and flat (`.event_ids`) paths:

```python
blocking = (
    _get(_get(stop_reason, "requires_action"), "event_ids")
    or _get(stop_reason, "event_ids")
    or []
)
```

  - If `blocking` is non-empty and is a subset of `pending_custom_tool_uses ∪ answered_custom_tool_uses`, continue the loop (we already responded or are about to — the union guards the race where a queued `status_idle` references an id we've answered).
  - If `blocking` is empty under `requires_action`, that's a schema-path miss — log the full raw `stop_reason` and raise SystemExit(1). Do NOT loop on empty blocking, or we'll hang until workflow timeout.
  - Otherwise, `blocking` has IDs we don't recognize (can happen when `status_idle` is delivered before its corresponding `agent.custom_tool_use` arrived on the stream). Before failing, **paginate** `client.beta.sessions.events.list(session.id)` until either every unknown id appears and is a custom_tool_use we can respond to (loop back and handle it) or the pagination is exhausted. Only raise SystemExit(1) after full pagination still leaves unknown blocking ids.
- any other (`retries_exhausted`, `max_turns`, `refusal`, `error`, …) → log full session + last events and raise SystemExit(1).

## Resolved Questions

- **Timezone**: Eastern Time, year-round — workflow uses `cron: '13 9 * * *'` with `timezone: 'America/New_York'` (GitHub Actions supports this; DST is handled automatically; `:13` dodges top-of-hour scheduler congestion)
- **GitHub username**: `ZH-L1N`
- **Slack duplicates on reruns**: **Allowed**. Every run (scheduled or `workflow_dispatch`) posts a fresh Slack message. No de-dup state tracking.
- **No-op reruns (byte-identical content)**: **Skip the commit, but still send Slack.** Verification is machine-checkable: the agent emits `CONTENT_SHA256: <hex>` in its final message (the single contract — no free-form sentinels). Orchestrator fetches today's file from GitHub, computes its SHA-256, and accepts the no-op only if the observed hash equals the agent-claimed hash. Slack always fires so the user sees the pipeline ran.

## Failure modes & diagnosis

- **GitHub MCP auth fails** (the `authorization_token` from the resource doesn't propagate to the MCP server as expected): the run exits non-zero via the `requires_action` / MCP error path. GitHub Actions surfaces the failed workflow. Diagnose by listing session events (`client.beta.sessions.events.list(session_id)`) and looking for `agent.mcp_tool_result` entries with `is_error`. No automatic fallback — if this fails repeatedly, switch the write path to an orchestrator-side `git` push instead of the agent-side MCP.
- **Session hangs at `requires_action` with no resolvable blocking IDs**: workflow `timeout-minutes: 15` kills the run; `run.py`'s robust extractor also raises SystemExit(1) proactively.
- **Slack webhook down**: agent logs the error via `is_error=true` tool result; `run.py` retries twice with 2s/4s backoff, then raises SystemExit(1) at end of loop so the workflow turns red.
- **Scheduled workflow auto-disabled**: GitHub automatically disables scheduled workflows on repos with no commit activity for 60 days. If the digest stops firing on schedule but `workflow_dispatch` still works, check **Actions → AI Daily Digest → Enable workflow**. Prevent by ensuring the orchestrator repo sees at least one commit every ~50 days (or wire a tiny scheduled no-op commit).
